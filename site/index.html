InspiTrip AI 旅行灵感推荐系统

面向“已经产生出行意愿、却说不清目的地”的旅行者。系统通过 AI 理解“想安静看海、不要太商业化”等模糊表达，将其转为可执行的 Query Plan，再由混合检索、事实约束、证据门控与确定性排序生成可解释的目的地建议。

An AI-assisted destination discovery product that turns vague travel feelings into constrained, evidence-backed and explainable recommendations.

一、文档概述

1.1 项目背景与用户痛点

传统 OTA 的搜索入口通常要求用户先给出目的地、日期或品类，但旅行灵感阶段的真实表达往往是“周末想松弛一点”“想看海，但不要太商业化”。用户不是没有需求，而是缺少把感觉翻译成搜索条件的能力。

表达断层用户说的是情绪、氛围和排斥项，平台需要的是地点、预算、天数和交通条件。
决策成本用户需要跨攻略、地图和评论反复比较，才能判断一个目的地是否真的适合。
信任缺口仅靠生成式文案容易把“听起来合理”误当成“事实可执行”。

1.2 为什么该场景适合使用 AI

这个场景同时包含语言歧义、长尾表达和语义匹配，适合让 AI 处理“理解与表达”；但预算、交通、证据和最终排序需要可复现，因此必须由规则、数据库和确定性代码接管。

AI 擅长理解模糊语言提取情绪、氛围、活动、否定项与潜在证据需求发现语义候选把口语化需求与目的地画像进行语义匹配组织自然表达基于已准入证据生成受限推荐理由
确定性系统负责核对硬条件预算、天数、交通方式与可达性不得由模型猜测执行排序策略感觉权重、证据门槛和 MMR 多样性均可复现控制失败状态缺字段则追问、无证据则降级、无结果则返回空状态
方案优势主要问题结论
纯标签筛选稳定、成本低难覆盖“松弛但不无聊”等组合语义只作为规则底座
端到端 LLM 推荐表达自然、搭建快事实幻觉、排序不可复现、难审计不采用
AI + 检索 + 规则兼顾语义理解与事实可信链路更长，需要分层评测当前方案

1.3 产品目标与目标用户

目标说明
用户目标降低表达和筛选成本，从一句自然语言获得愿意继续了解的候选目的地。
产品目标在预算、天数和交通可执行的前提下，提高“想去 / 收藏 / 明确正反馈”的有效灵感转化。
质量目标硬约束可验证、推荐理由有证据、失败状态透明，不用幻觉填满结果。
目标用户已有出游动机但目的地未定、习惯用感受描述需求、希望降低攻略搜索成本的旅行者。
北极星指标：有效灵感转化率 = 有“想去 / 收藏 / 明确正反馈”的需求会话 ÷ 已提交旅行推荐需求的会话。

二、功能需求

本版本覆盖“灵感输入 → 渐进澄清 → 意图结构化 → 候选召回 → 过滤排序 → 证据输出 → 反馈回流”的完整推荐链路。点击模块可查看左侧对应交互。

2.1 灵感输入与输入契约

功能描述：允许用户先用一句自然语言表达旅行感觉，再补充出发地、预算与天数。自然语言承担开放表达，表单承担高风险事实输入。

输入类型设计目的
query必填文本承载情绪、氛围、活动、排斥项与自由表达
origin条件必填决定交通可达性，是当前唯一阻断槽位
budget / days可选数字用于硬过滤；缺失时允许带假设推荐
conversation_id可选标识支持多轮修改与 Query State 延续

2.2 渐进澄清

核心逻辑：默认优先生成结果，只在缺少会阻断推荐的字段时追问一次。出发地缺失时询问“从哪座城市出发”；预算和天数缺失时不连续盘问，而是在结果中标记假设。

2.3 Query Plan 意图结构化

输出结构：将原始 Query 转为 scope、task_type、hard_constraints、exclusions、semantic_query、soft_preferences 和 evidence_aspects。Schema 禁止额外字段，标签带置信度，否定项与正向偏好分离。

2.4 混合候选召回

功能描述：Dify 根据 semantic query 执行 BM25、Dense Retrieval 与 Reranker，返回 destination_id + semantic score。检索只负责发现候选，不直接决定最终展示顺序。

2.5 事实过滤与确定性排序

核心逻辑：后端从 PostgreSQL 或 JSONL 补齐目的地事实，依次执行预算、天数、交通硬过滤，mood / vibe / activity 加权排序，证据准入和 MMR 多样性选择。相同输入可得到可复现结果。

2.6 证据推荐与输出契约

输出字段：kind 区分消息、推荐与空结果；answer 给出状态说明；recommendations 包含名称、城市、受限理由、事实详情、分数与合成数据标识；同时返回 query_plan 和候选诊断信息。

Entity地点身份
Evidence来源凭证
Claim原子判断
Fact预算与交通
Profile感觉画像

2.7 产品状态与反馈闭环

状态覆盖：正常、生成中、待澄清、空结果和服务降级。结果卡支持证据展开以及“想去 / 不感兴趣 / 信息有误”反馈，分别进入转化指标、排序诊断和数据纠错队列。

2.8 失败与降级

设计原则：失败时优先保留事实可信度，而不是保留卡片数量。证据不足的候选不输出；条件过严时返回空结果和调整建议；外部服务不可用时展示可重试状态，不用模型生成未经核实的替代答案。

三、技术架构与选型

选型原则是“AI 处理语义，规则控制风险，数据提供事实”。公开 Demo 保持零密钥运行；完整模式保留真实编排、数据库、地图与模型服务的接入边界。

层级选型用途选型理由

API 与契约FastAPI + PydanticChat、Ranking、Location、Analytics类型清晰、Schema 可验证、便于拆分测试

3.1 关键架构决策

  • Workflow 而非 ReAct Agent：旅行推荐包含事实、预算和证据边界，固定工作流能限制调用次数、延迟与不可预测行为。
  • 检索与排序解耦：Dify 只返回候选 ID 和语义分数；后端负责事实补齐、过滤与最终选择。
  • 理由与事实分离：LLM 只生成受限理由；预算、交通、限制和降级信息由代码渲染。
  • 双运行模式：demo 使用合成 JSONL 和真实排序逻辑，full 接入 Dify、PostgreSQL 与外部服务。

四、调用流程

4.1 单次推荐核心流程

  1. 接收需求前端提交 query、origin、budget、days 与 conversation_id。
  2. 校验与澄清Pydantic 校验请求；缺少出发地时返回 needs_clarification，不启动推荐。
  3. 生成 Query Plan模型输出通过 JSON Schema 校验；异常时使用规则解析兜底。
  4. 召回候选Dify 执行 BM25、Dense 与 Reranker，输出 destination_id 和语义分数。
  5. 补齐事实Repository 从 JSONL / PostgreSQL 读取地点、预算、交通、画像与 Claim。
  6. 过滤与排序硬约束过滤 → 感觉加权 → Claim 重排 → Evidence Gate → MMR。
  7. 校验输出理由只能引用已准入 Claim，事实字段由代码卡片渲染。
  8. 反馈回流记录曝光、证据展开和反馈事件，支持排序诊断与实验分析。

4.2 输入输出设计

REQUEST用户可控输入query · origin · budget · days自然语言负责表达，结构化字段负责高风险约束。
→
QUERY PLAN机器可执行意图hard_constraints · soft_preferences · evidence_aspects把语义理解与推荐执行解耦。
→
RESPONSE可验证结果kind · recommendations · diagnostics明确区分推荐、追问、空结果与错误状态。

4.3 失败场景与兜底策略

失败点系统判断用户侧兜底
缺少出发地无法判断可达性只追问一次关键字段，不进入错误推荐
模型输出不符合 SchemaQuery Plan 不可执行使用规则解析 fallback，并保留原始输入
候选不满足硬约束可用结果为 0返回空结果，指出应放宽的预算、天数或活动条件
Claim 证据不足理由缺少支持候选降级或移除，不生成看似合理的文案
Dify / 外部服务异常完整链路不可用展示服务降级与重试入口；公开环境仍可使用 Demo
事实相互冲突结果不可信保留限制提示，进入“信息有误”数据治理队列
兜底原则：AI 可以少给答案，但不能用未经验证的答案填补失败。

5.2 怎么评估 AI 是否有效

01 · 解析Query Plan槽位、否定、置信度、多轮修改与 Schema 合法性
02 · 召回RetrievalRecall@K、Hit Rate、MRR 与库存覆盖
03 · 决策Ranking硬约束误杀、感觉匹配、证据准入与多样性
04 · 输出Faithfulness事实忠实度、引用完整性、空结果与降级状态
05 · 业务Conversion有效灵感转化、错误率、空结果率、P95 与用户反馈

线上效果不以“模型回答次数”判断，而以有效灵感转化率为北极星；“想去”计入正向转化，“不感兴趣”用于排序诊断，“信息有误”进入数据治理。首个 A/B 实验比较定位权限在页面加载时触发，还是在用户表达出行意图后触发。

自动化 Smoke Test 仅产生两条 Demo 会话。由此得到的 50% 转化率只能证明埋点与计算链路可用，不能作为真实实验结论。

六、范围与边界

6.1 本版本包含

  • 目的地发现、一次渐进澄清、Query Plan、混合召回、硬约束过滤与感觉排序。
  • Claim 级证据门控、受限推荐理由、事实卡、空结果、服务降级与反馈闭环。
  • 无密钥 Demo 模式，以及 Dify / PostgreSQL / 高德 / 模型服务的完整模式接入边界。

6.2 明确不做

  • 交易履约：不包含酒店、机票、门票预订和支付。
  • 自动执行型 Agent：不让模型自主决定调用链或替用户下单。
  • 无证据自由生成：没有 Claim 支持时不生成确定性推荐理由。
  • 完整实时交通：高德路线估算不等于铁路、轮渡班次与票务信息。

6.3 当前能力边界

AI 负责“理解感觉、发现候选、组织受限表达”；确定性系统负责“核对事实、执行约束、选择结果”；用户负责“确认真实偏好并做最终旅行决策”。
  • 公网页面是三个引导式合成场景，不把固定结果伪装成实时 AI 自由生成。
  • Query State 当前为进程内存储，多 worker 生产部署需要迁移到共享存储。
  • Dify blocking 链路 P95 偏高，需要流式响应与节点级性能优化。
  • 早期 Extraction 数据仍有兼容迁移层，生产化需进行 Schema 版本化重抽。
